AIMS Racket Assignment Questions

Please read this section completely before attempting any assignment questions. Links to individual questions are in the sidebar to the left. Some questions may require Internet access to properly display embedded mathematics; if there is no Internet access, the mathematical notation should still be readable in text form.

All questions are to be done individually unless otherwise specified. Please do not consult books, papers, or Web pages. Do not look at the code of other students, or show your code to other students. You may discuss assignment questions with other students at a high level, but please do not take any notes during these discussions. If you run into difficulties, ask me (the instructor) or the tutors for help.

When I teach this material over twelve weeks in Canada, I use a server to which students submit assignment solutions. The server tests student programs using test expressions that are unknown to the students. In a few minutes after submission, the student receives a report that either says that their program passed all tests, or lists the numbers of the first two tests that failed. This does not provide much information. It only says that the submitted program is not fully correct. Students are supposed to write their own tests. Furthermore, each student can only submit three times in any 24-hour period.

Because the AIMS course is only three weeks long, and because the server is located in Canada, I am not using it. Instead, for each assignment question, I have written a evaluation program containing the test expressions. You will be provided with a copy of this program, which has been compiled so that it is no longer a text file, and the test expressions are not visible. Please do not try to discover the test expressions, or use this file in any way other than as described below. Note that the evaluation program only checks this limited form of technical correctness. Some questions ask you to follow a certain style, or to use only certain features of Racket. The evaluation program will not enforce this. You must submit a final version of your program to the tutors before the assignment deadline, and the tutors may deduct marks if your program fails to meet the specifications in the question, even if it is technically correct.

Lecture module 02 describes the design recipe, a method of designing and testing functions. Please follow this for all assignment questions. Not only does this help you to do your work, it provides us with valuable information in the case that you come to us for assistance. Once you have thoroughly tested your assignment solution, you can check it against the evaluation program as follows. (The example below is for Question 1; simply change the number for subsequent questions.)

Your solution to Question 1 will be in the file Q1.rkt. The associated testing program is called Q1-test_rkt.zo. Move this file into the same folder as your solution. Also move the file Q1-eval.rkt into this folder. This is the evaluation program. Then use DrRacket to open Q1-eval.rkt. You should see the following two lines:

(require "Q1-test_rkt.zo")
(run-tests)

Run Q1-eval.rkt in DrRacket. You should get a report in the Interactions window. Unlike with the Canadian server described above, you will see the results of all tests. There is also no restriction on how often or how frequently you can run a testing program. Please try to use it as little as possible. When it says you have an error, think carefully about your program, and write more tests of your own, rather than quickly making a change and then trying the evaluation program again.

Here are the possible test results reported by the evaluation program:

"Success: test passed". You can congratulate yourself for this one.

"Error: wrong result for test". This usually means that our test expression did not produce the expected value when evaluated using your program. It can also happen with some assignment questions if your program was supposed to deliberately raise an error with my test expression, but it did not.

"Error: program ran out of resources while running test". This means that your program took too much time or too much memory during the test.

"Error: program encountered error while running test". This means that your program halted with an error during the evaluation of a test expression, but it wasn’t supposed to. One very simple reason could be that you have made a spelling mistake, so the function expected by the test expression is not defined in your program. This error can also happen if the evaluation of the test expression was supposed to halt with an error or raise an error, and it did, but the error message did not match the assignment specification.

"Error: program ran out of time while initializing". This happens if your program takes too much time when it is evaluated by the testing program, before any tests are run.

"Error: program ran out of memory while initializing". Same as above, but with memory. This is rare.

"Error: program failed to initialize". This can happen if your program has a runtime error when run in DrRacket. If that doesn’t happen when you run your program yourself, it usually means that you have used the wrong language level.
